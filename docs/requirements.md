## 1. 프로젝트 목표

AWS ALB 리소스(약 60개)와 Route53 레코드를 수집, 분류, 정리하기 위한 도구를 개발한다.
- 불필요한 ALB를 식별하여 삭제 또는 통합(합치기)한다
- 솔루션(프로젝트)별로 ALB를 정리한다
- 웹 UI를 통해 현황을 공유하고 합치기 대상을 지정한다

## 3. 기능 요구사항

### 3.1 데이터 수집 (Collector)
- **FR-01**: AWS ALB 목록 수집 (ELBv2 API)
- **FR-02**: k8s로 시작하는 ALB 이름 자동 제외
- **FR-03**: Route53 레코드 전체 조회 후 메모리에 적재, ALB DNS와 매칭 (개별 조회 금지)
- **FR-04**: Target Group 상태 수집 (healthy, unhealthy, 타겟 없음)
- **FR-05**: Go goroutine 기반 병렬 수집
- **FR-06**: 수집 결과를 JSON 파일로 저장 (`data/entries.json`)

### 3.2 분류 (Classifier)
- **FR-07**: ALB 이름 패턴으로 솔루션(프로젝트) 및 환경 자동 매핑
  - ALB 네이밍 패턴: `{aws리소스}-{환경}-{솔루션}-{서비스선택}` (예: `alb-dev-signage-xxx`)
  - 환경 식별: 패턴의 두 번째 세그먼트로 dev/stg/prd 등을 판별
- **FR-08**: 조치상태 자동 추론 규칙:
  - Target Group이 없는 ALB → 삭제 후보
  - Route53 레코드에 연결되지 않은 ALB → 삭제 후보
  - Route53에 연결되어 있지만 에러 응답(500대, 인증서 에러 등) → 공지 후 삭제 (필요 시 합치기)
  - **응답 판정 기준**: 200~499 = 정상 (400대는 서버가 응답 중이므로 정상), 500+ = 에러. 인증서 에러/타임아웃도 에러로 통합 처리.
  - **에러 확인 방법**: Route53 레코드 값(도메인)으로 HTTPS(443) 요청을 보내서 응답 코드 확인. ALB DNS 직접 요청이 아니라 Route53 레코드에 연결된 도메인으로 요청한다. 인증서 에러도 "에러"로 통합 처리한다.
  - **요청 경로**: 루트(`/`)를 먼저 시도하고, `/health`, `/healthz` 등 일반적인 헬스체크 경로도 추가 시도한다. 모든 경로에서 에러 응답 시 "에러"로 분류한다.
- **FR-09**: 자동 분류 불가 항목은 사용자에게 티키타카 질의
  - 네이밍 패턴 불일치 ALB: 패턴 `{aws리소스}-{환경}-{솔루션}-{서비스선택}`을 따르지 않는 ALB도 존재하며, 이 경우 사용자에게 질의
  - 솔루션 선택: "이 ALB의 솔루션을 선택해주세요: 1. signage, 2. cms ..."
  - 환경 선택: "이 ALB의 환경을 선택해주세요: 1. dev, 2. stg, 3. prd"
  - 조치상태 선택: "이 ALB의 조치상태를 선택해주세요: 1. 유지, 2. 합치기, 3. 삭제"
- **FR-10**: 분류 결과를 JSON에 반영

### 3.3 합치기 (Merge) 계획
- **FR-11**: 환경-솔루션 단위로 ALB 통합 (예: `alb-dev-signage` 하나로 dev 환경의 signage 관련 ALB를 통합)
- **FR-12**: 합치기 시 필요한 정보 표시:
  - 이동해야 할 Route53 레코드 목록
  - 이동해야 할 리스너 룰 목록
  - Target Group 재연결 정보
- **FR-13**: 본 도구는 "계획/시각화"까지만 담당, 실제 AWS 리소스 변경은 별도 절차

### 3.4 웹 UI
- **FR-14**: ALB 현황 테이블 뷰
  - 열: ALB 이름, 솔루션(프로젝트), 이용상태(정상/에러/타겟없음 등), 조치상태(유지/합치기/삭제)
- **FR-15**: 솔루션별, 조치상태별 필터링
- **FR-16**: 드래그앤드롭으로 합칠 대상 ALB 지정
  - 합칠 대상 ALB 이름을 만들고, 드래그하면 합쳐질 ALB 정보가 매핑됨
  - 매핑 시 레코드 정보, ALB 정보 등이 표시됨
- **FR-17**: 임시용 도구이므로 단순 HTML/JS 단일 파일
- **FR-18**: 데이터는 파일 시스템 기반 (DB 사용 안 함)
- **FR-19**: 웹 UI는 localhost에서만 접근 (별도 서버 배포 불필요)

```json
  "albName": "string",

  "environment": "dev | stg | prd | unknown",
  "status": "healthy | unhealthy | no_target | no_record | error | unknown",
  "action": "유지 | 합치기 | 삭제 | 미정",

      "name": "string (Route53 레코드 이름)",

      "type": "ALIAS | CNAME",

  "targetGroups": [
    {
      "name": "string (Target Group 이름)",
      "arn": "string",
      "healthyCount": "int",
      "unhealthyCount": "int",
      "unusedCount": "int",
      "targetCount": "int (등록된 타겟 총 수)"
    }
  ],
  "mergeTarget": "string | null",
  "note": "string"

```

## 5. 조치상태 분류 기준

| 조치상태 | 조건 |
|----------|------|
| 삭제 | Target Group 없음, 또는 Route53 레코드 미연결, 또는 에러 응답(500/인증서 에러 등), 또는 중지된 프로젝트(컨펌 후) |
| 합치기 | 같은 환경-솔루션의 다른 ALB에 통합 가능, 또는 중지예상 프로젝트(이어받은팀 ALB로) |
| 유지 | 정상 운영 중이며 통합 불필요 |
| 미정 | 자동 판단 불가, 사용자 입력 대기 |

## 6. 기술 제약사항

- **TC-01**: Go 언어 사용 (병렬 처리 필요)
- **TC-02**: AWS CLI 이미 연결됨, 별도 credential 설정 불필요
- **TC-03**: AWS SDK v2 사용
- **TC-04**: API rate limit 대응 (병렬도 조절, 재시도 로직)
- **TC-05**: 파일 기반 데이터 저장 (JSON)

---

## 7. 비기능 요구사항

- **NFR-01**: 수집 시 ALB 60개 + 레코드 다수를 합리적 시간 내에 처리
- **NFR-02**: 임시용 도구이므로 과도한 설계 지양
- **NFR-03**: 실제 AWS 리소스 변경(삭제, 합치기 실행)은 본 도구의 범위 밖

## 8. 확인 완료 사항

- [x] **Q1**: 에러 판단 기준 → Route53 레코드 값(도메인)으로 HTTPS 요청을 보내서 응답 코드 확인. ALB DNS가 아닌 Route53 레코드에 연결된 도메인으로 요청.
- [x] **Q2**: "환경" 분류 → ALB 네이밍 패턴 `{aws리소스}-{환경}-{솔루션}-{서비스선택}` 에서 두 번째 세그먼트로 환경 식별. 예: `alb-dev-signage-xxx`
- [x] **Q3**: 웹 UI 접근 → localhost에서만 사용. 별도 서버 배포 불필요.
- [x] **Q4**: HTTP 프로토콜/인증서 → HTTPS(443)로 요청. 인증서 에러도 500 에러와 동일하게 "에러"로 통합 처리.
- [x] **Q5**: 네이밍 패턴 불일치 → 패턴을 따르지 않는 ALB도 일부 존재하며, 이 경우 사용자에게 티키타카 질의로 솔루션/환경을 확인.
- [x] **Q6**: 요청 경로 → 루트(`/`) + `/health`, `/healthz` 등 일반적인 헬스체크 경로도 시도. 모든 경로에서 에러 시 "에러"로 분류.
